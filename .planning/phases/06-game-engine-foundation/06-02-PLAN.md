  <behavior>
**Timing Window (GAME-01):**
- Plugin's `setCurrentTrack` now accepts featurings and malus terms
- Plugin tracks `windowOpenAt: number | null` in TrackState
- `windowOpenAt` is set to `Date.now()` on the first correct title or artist guess
- Plugin exposes `getWindowOpenAt()` so BotSession can compute decay externally
- Plugin does NOT compute decay points itself — it returns base points (3 for first, or a flag for "in window"). The BotSession layer computes the actual decayed points.
- ALTERNATIVELY (simpler for TDD): the plugin receives a `now` timestamp in the ChatMessage (or a separate param) and computes decay internally. This keeps the plugin self-contained for testing.
- **Decision: Plugin computes decay internally.** `setCurrentTrack` receives `windowDurationMs` parameter. `onChatMessage` uses `message.timestamp` as the clock. `windowOpenAt` is set to the timestamp of the first correct guess. Decay formula: `pts = 3 - 2 * (elapsed_ms / window_ms)` rounded to 1 decimal. Post-window returns null.

**Malus (GAME-03):**
- `setCurrentTrack` receives `malusTerms: string[]` (loaded from playlist)
- Before checking title/artist, check if message fuzzy-matches any malus term (tolerance 0.15)
- If malus matches: return ScoringEvent with reason='malus', points=-N (N = counter for this viewer this round), add viewer to a malus-locked set (viewer cannot guess correctly this round after malus)
- malusCounters: Map<viewerUsername, number> in TrackState, incremented per malus hit
- If message matches both malus AND correct answer: malus wins (return malus event only)

**Double-Shot (GAME-04):**
- When message matches BOTH title AND artist:
  - Compute title_pts and artist_pts using decay formula (or 3 if first finder)
  - Return (title_pts + artist_pts) x 2 with reason='double_shot'
  - If only ONE of the two matches in a "long enough" message: this is NOT a double-shot attempt, just a normal single match
  - WAIT — per CONTEXT.md, double-shot is when BOTH are correct. If only one is correct, the viewer gets 0 for both. But how does the plugin know the viewer intended a double-shot vs a normal guess? Answer: if the message matches both title AND artist simultaneously (fuzzyMatch on both), it's a double-shot. If it matches only one, it's a normal guess (scores normally for that one answer). The "0 for both if only one correct" applies only to double-shot ATTEMPTS — but since we can't detect a "double-shot attempt" vs a normal guess, the rule is: if both match, it's a double-shot with bonus; if only one matches, it's a normal guess.
  - CORRECTION based on CONTEXT.md careful re-read: "Viewer can submit title + artist in one message (double-shot): both correct = bonus points; only one correct = zero points for both." This means: if a viewer's message matches title but NOT artist (or vice versa), and the message ALSO attempted the other (i.e., the message contains words that are not matching), the viewer gets 0. But we can't detect "attempted the other" reliably. The simplest interpretation: if a message matches ONLY title, score normally as correct_title. If it matches ONLY artist, score as correct_artist. If it matches BOTH, score as double_shot with bonus. The "0 for both" scenario cannot trigger in the plugin alone — it would require BotSession-level logic to detect a "double-shot attempt." For now, implement the bonus path only. Document this decision.

**Featurings (GAME-05):**
- `setCurrentTrack` receives `featurings: string[]`
- TrackState tracks `foundFeaturings: Set<string>` (which featurings already found)
- Featuring check: after title/artist check (only if viewer hasn't already scored title/artist this round), check if message fuzzy-matches any unfound featuring
- Return ScoringEvent with reason='featuring', points=1
- No timing window for featurings
- Featuring does NOT add viewer to answeredViewers (viewer can still guess title/artist after guessing a featuring)
